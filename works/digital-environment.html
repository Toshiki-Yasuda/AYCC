場所を選ばない開発体制

クラウド・モバイル・AIを組み合わせた「切れない」業務環境

この環境を構築した背景

AYCCのエンジニアは、システム開発業務と並行して家族の介護を担っています。作業場所は自宅のデスクに限らず、病院の待合室や移動中の電車内など日々変化します。

病院ではいつ診察に呼ばれるか分かりません。「端末を閉じて即座に移動し、戻ったらすぐ作業を再開する」。この要件を満たすために、場所や中断に左右されない開発環境を設計・構築しました。

結果として、お客様へのレスポンスや納品スケジュールに影響を出さない体制が実現しています。

「どこでも同じ環境」を実現する仕組み

この開発体制の基本的な考え方は、手元のデバイスに開発環境を持たないことです。

ソースコード、データベース、AIツールなど開発に必要なすべてのリソースはクラウド上のサーバー（VPS）に集約されています。自宅のPC、外出先のMacBook、病院でのiPad mini、いずれもこのサーバーへの「窓口」に過ぎません。そのため、どのデバイスから接続しても、まったく同じ作業環境にアクセスできます。ローカルにデータを保存しないため、端末の紛失・故障時にも業務データが失われるリスクがありません。

構成イメージ

Windows（自宅） MacBook（外出先）
iPad mini（病院など） Android（空き時間）

すべて同一サーバーに接続

VPS（クラウド上の開発環境）

デバイスごとの役割の違いは、画面サイズと使用場面だけです。

  • Windows デスクトップ（自宅）

    大画面で複数ウィンドウを並べた設計作業や、負荷の高い処理に適しています。

  • MacBook Pro（外出先）

    カフェなど腰を据えて作業できる場面で使用。コーディングやコードレビューに活用しています。

  • iPad mini（病院など身軽な外出時）

    文庫本とほぼ同じサイズ。病院の待合室や電車内など、PCを広げにくい場面でも周囲に圧迫感を与えず、短時間の作業やお客様とのやり取りに使用しています。

  • Android端末（空き時間）

    Termux + Moshにより、スマートフォンからもVPSに直接接続。空き時間にコーディングの進捗チェックや軽微な修正を行えます。

中断に強い通信技術

クラウド上の環境を活かすには、場所や状況を問わず安定して接続できる仕組みが必要です。AYCCでは、一般的なリモート接続よりも中断に強い技術を組み合わせています。

Mosh（Mobile Shell）：中断に強いリモート接続

一般的なSSH接続は、通信が途切れるとセッションが切断されます。Moshは、UDPベースの独自プロトコルにより、ネットワークの切断・再接続を自動でハンドリングします。

地下鉄での通信途絶、Wi-Fiからモバイル回線への切り替え、端末のスリープといった場面でも、接続状態を維持します。端末を開けば、前回の作業状態がそのまま表示されます。

技術的には、SSHが「電話」のように常時接続を前提とするのに対し、Moshは「メッセージアプリ」のように断続的な通信でも成立する仕組みです。

tmux：複数作業の並行管理

VPS上ではtmux（ターミナルマルチプレクサ）を使用し、複数の作業画面を常時維持しています。コーディング・テスト実行・AI連携・ログ監視などを個別のウィンドウで並行管理できます。

tmuxのセッションはサーバー側で保持されるため、接続元のデバイスが変わっても、すべてのウィンドウ配置と作業状態がそのまま引き継がれます。

最新AIモデルに最適化した開発体制

AYCCでは当初、特性の異なる複数のAI CLIツールを使い分ける「マルチCLI」体制で開発を行っていました。しかし2026年2月6日、Anthropic社の最新モデルClaude Opus 4.6のリリースを機に、開発環境を根本から刷新しました。

Opus 4.6は、従来は複数のAIに分担させていた設計・実装・レビューを単一モデルの中で並行処理できるサブエージェント機能を備えています。これにより、モデル間のコンテキスト受け渡しロスがなくなり、品質と速度の両方が向上しました。

現在の開発体制

Anthropic Claude Code Opus 4.6

サブエージェントによる並行処理

設計・計画 実装・コード生成 レビュー・検証

コンテキストの一貫性

プロジェクト全体の文脈を保持したまま、設計から実装・テストまでを一貫して処理。複数モデル間での情報の欠落がありません。

自律的な並行処理

サブエージェント機能により、独立したタスクを同時に実行。調査と実装の同時進行など、チーム開発に近い動きを1モデルで実現します。

高度な判断力

Opus 4.6の深い推論能力により、設計判断やエッジケースの検討をAI自身が行い、エンジニアは最終確認に集中できます。

小さな画面のiPad miniからでも高度な開発が可能なのは、日本語による指示でAIが自律的に作業を進め、エンジニアは方針決定と品質管理に集中できる体制だからです。

ワークフロー例

エンジニアが方針指示 → Opus 4.6 が設計・実装・検証を並行処理 → エンジニアが最終判断

最終的な品質判断と方針決定は必ず人間が行います。AIの出力を盲信するのではなく、エンジニアの経験に基づく判断を最上位に置いています。

CLAUDE.md によるプロジェクト管理

AIの動作は場当たり的に制御しているわけではありません。各プロジェクトのルートに配置するCLAUDE.mdファイルに、コーディング規約・アーキテクチャ方針・禁止事項・ワークフローを記述しています。

Opus 4.6はこのファイルを自動的に読み込み、プロジェクト固有のルールに従って自律的に判断・行動します。エンジニアが毎回細かく指示を出さなくても、一貫した品質基準とコードスタイルが保たれる仕組みです。

CLAUDE.md（方針・規約） → Opus 4.6 が自律判断 → 並行実行・品質検証 → エンジニアが承認

従来の独自フレームワーク（ADS）は廃止し、Claude Code自体が持つサブエージェント機能とCLAUDE.mdの組み合わせに移行しました。ツール側の進化に合わせて常に最適な体制を追求しています。

お客様にとっての意味

この体制により、以下のようなサービス品質を維持しています。

  • 迅速なレスポンス

    場所を問わず業務環境にアクセスできるため、お問い合わせや緊急対応にも柔軟にお応えできます。

  • 安定した納品スケジュール

    作業の中断・再開が即座に行えるため、細切れの時間も有効に活用し、スケジュール遅延のリスクを低減しています。

  • AI活用による開発効率

    複数AIの連携により、個人事業主でありながらチーム開発に近い生産性と品質チェック体制を実現しています。
